{% extends "app_base.html" %} {% load i18n core_tags %} {% comment %} Validibot override of allauth's `mfa/totp/activate_form.html`. Renders the TOTP activation flow (QR code + secret + verification code) inside our in-app chrome (left nav, top-bar breadcrumbs) with a "Back to Security" affordance. Why override the allauth leaf directly instead of allauth's base_manage? Django template block inheritance doesn't compose cleanly through allauth's base_manage layer when the leaf templates override `{% block content %}`. We render the allauth form directly inside `app_base.html`, matching the pattern used by `users/security.html`. The form POSTs to the same URL as allauth's upstream (`mfa_activate_totp`), and the visible field (`code`) comes from the allauth `ActivateTOTPForm` that allauth's view provides. {% endcomment %} {% block title %} {% trans "Activate Authenticator App" %} {% endblock title %} {% block bodyclass %} {{ block.super }} account-settings-page {% endblock bodyclass %} {% block left_sidebar %} {% include "app/partial/components/app_left_nav.html" %} {% endblock left_sidebar %} {% comment %} Populate the top-bar breadcrumb slot. `app_top_bar.html` includes `core/partial/breadcrumbs.html`, which reads `breadcrumbs` from context. Our own Django views set that via `BreadcrumbMixin`, but allauth views don't — so we build it in-template via the `mfa_breadcrumbs` tag and re-include the top bar inside the `{% with %}`. {% endcomment %} {% block top_bar %} {% trans "Activate Authenticator App" as leaf_label %} {% mfa_breadcrumbs leaf_label as breadcrumbs %} {% include "app/partial/components/app_top_bar.html" %} {% endblock top_bar %} {% block app_content %}
